# UAT — User Acceptance Testing

This folder contains everything for the UAT workflow run by the `uat-agents` Claude Code plugin: planning, reviewing, executing tests, logging bugs, and accumulating UX observations. It is self-contained and can be published independently of the rest of the repository.

The `uat-agents` plugin is **not** vendored here — only its output is. Templates and skill definitions ship with the plugin and live under `${CLAUDE_PLUGIN_ROOT}/` at install time.

## How to read this folder

- **`ux-report.md`** — cumulative idea repository. Tester appends to it on clean session completion, deduplicating against existing entries. Curate it and use it as input for future improvement phases.
- **`sessions/`** — one folder per planning + test cycle, named `YYYY-MM-DD-{slug}`.
- **`sessions/<slug>/plan.md`** — approved test plan. Status field gates execution.
- **`sessions/<slug>/review.md`** — auto-generated by the reviewer subagent.
- **`sessions/<slug>/session.md`** — tester's run state (resume contract).
- **`sessions/<slug>/bugs/`** — one `BUG-NNN.md` per bug, plus `INDEX.md` and `evidence/`.

## Commands

| Command | Purpose |
|---|---|
| `/uat-agents:plan` | Author a plan; auto-runs the reviewer; you approve manually. |
| `/uat-agents:plan-review` | Re-run the reviewer after hand-editing a plan. |
| `/uat-agents:test <slug> [resume\|restart\|reverify\|subset]` | Execute an approved plan. |

Run `/clear` between commands so each agent gets a fresh context.

## End-to-end workflow

```
1. /uat-agents:plan
   ├─ Q&A with you
   ├─ Drafts plan.md (status: draft)
   ├─ Auto-runs reviewer subagent → review.md (status: in-review)
   ├─ Auto-fixes unambiguous BLOCKERS, surfaces the rest
   └─ You approve → status: approved

2. /clear

3. /uat-agents:test <slug>
   ├─ Verifies plan is approved
   ├─ Runs tests via Chrome extension
   ├─ Logs each pass/fail; writes bugs on failures
   ├─ On critical bug: ABORT, point you at the debug agent
   └─ On clean finish: consolidate UX observations into ux-report.md

4. (if aborted) /clear → invoke debug agent against the critical bug

5. /clear → /uat-agents:test <slug>            # default: resume
   or       /uat-agents:test <slug> reverify   # re-run from start to confirm fix
```

## Severity rubric (default)

| Tier | Definition | Tester action |
|---|---|---|
| **critical** | Crash, hang, infinite loop, unresponsive UI, page won't load, data loss | Abort the session |
| **high** | Feature broken; this test cannot pass | Log; skip strictly-dependent tests; continue |
| **medium** | Incorrect behavior with workaround, or non-blocking spec violation | Log; continue |
| **low** | Cosmetic, copy issues, minor polish gaps | Log; continue |

You can override the rubric per session in the plan's `severity_overrides` frontmatter field.

## Approval gate (hard rule)

The tester refuses to run any plan whose `status` field is not exactly `approved`. This is intentional: the review phase is the safety net for ordering bugs and unmet preconditions.

If you hand-edit an approved plan, run `/uat-agents:plan-review` — it demotes the plan to `in-review` and forces a fresh review pass.

## Conventions

- **Test IDs (`T-NNN`) are stable.** Once assigned, never renumber. New tests get new IDs.
- **Bug IDs (`BUG-NNN`) are session-scoped.** Sequential within a session.
- **UX-report IDs (`NAV-014`, etc.) are global and append-only.** They live across sessions.
- **One observable behavior per test.** The reviewer flags multi-assertion tests; consider splitting.
- **Happy path before edge cases within a group**, so an early abort still validates the critical path.
